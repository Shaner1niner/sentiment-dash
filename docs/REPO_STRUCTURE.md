# Repo Structure

This document explains the current `sentiment-dash` repository structure for Dashboard v2 work.

The repo is organized around a static dashboard architecture. The browser loads HTML/CSS/JS assets, the dashboard JavaScript reads generated JSON payloads, and local refresh scripts rebuild those payloads from the enriched SETA data pipeline.

## High-level architecture

```text
local data/export pipeline
  -> enriched chart-history CSV
  -> SETA screener / indicator / archetype outputs
  -> public/member chart JSON payloads
  -> static dashboard HTML/CSS/JS
  -> public/member dashboard views
  -> public-safe context cards and website snippets
```

This is not currently a React/Vite/Next application. Treat it as a static dashboard and payload repository unless a future phase intentionally migrates the structure.

## Root-level dashboard files

| File | Role |
| --- | --- |
| `interactive_dashboard_fix24_public_embed.html` | Public dashboard shell. Loads Plotly, shared CSS, and `dashboard_fix26_app.js`. Sets public mode defaults. |
| `interactive_dashboard_fix24_member_embed.html` | Member dashboard shell. Uses the same app engine with broader member controls/assets. |
| `interactive_dashboard_fix24_externalized_loader.html` | Externalized loader variant if present/active. Verify current use before editing. |
| `dashboard_fix26_app.js` | Main dashboard engine: manifest loading, chart rendering, controls, Market Tape/screener integration, alerts, panes, hovers, and annotations. |
| `dashboard_fix26_base.css` | Shared dashboard styling. |
| `dashboard_fix26_mode_manifest.json` | Public/member mode contract: titles, subtitles, data URLs, asset lists, control defaults, visible controls, badge order, legend and lower-pane behavior. |
| `build_fix26_chart_store_payloads.py` | Builds lean public/member chart-store JSON payloads from enriched chart-history CSV using the manifest asset lists. |
| `refresh_fix26_dashboard_all.bat` | Windows refresh workflow for exporting data, building screener outputs, building JSON stores, running smoke tests, staging changes, and optionally committing/pushing. |
| `SMOKE_TEST.md` | Legacy/partial smoke-test notes. The Python smoke test is the stronger current validation point. |
| `.gitignore` | Ignores local scratch, patch scripts, backups, logs, draft/runtime outputs, and public content archives while preserving latest public snippets. |

## Generated dashboard payloads

These files are generated by the refresh/build workflow and consumed by the dashboard.

| File | Consumer | Notes |
| --- | --- | --- |
| `fix26_chart_store_public.json` | Public dashboard mode | Generated public chart payload. Asset list comes from `dashboard_fix26_mode_manifest.json`. |
| `fix26_chart_store_member.json` | Member dashboard mode | Generated member chart payload with broader asset coverage/history. |
| `fix26_screener_store.json` | Market Tape / screener views | Generated SETA screener payload. Expected by `dashboard_fix26_app.js` and smoke-tested by `scripts/smoke_fix26_dashboard.py`. |

Generated JSON should not be hand-edited except for emergency diagnosis. Regenerate through the builder/refresh workflow instead.

## Scripts directory

| Path | Role |
| --- | --- |
| `scripts/smoke_fix26_dashboard.py` | Main dashboard smoke test. Checks screener store structure, chart payload presence/size, expected dashboard JS hooks, and embed cache tokens. |
| `scripts/build_seta_market_screener.py` | Builds SETA market screener, indicator matrix, and signal archetype outputs when versioned in this repo. The refresh script can fall back to a local copy. |
| `scripts/build_fix26_screener_store.py` | Builds `fix26_screener_store.json` for website/dashboard consumption. |
| `scripts/smoke_seta_public_context_cards_v2.py` | Public context card smoke test when present. Checks page hooks and public-safe snippet invariants. |

## Public content and context cards

| Path | Role |
| --- | --- |
| `seta_public_context_cards.html` | Standalone page rendering public-safe SETA context cards from `public_content/seta_website_snippets_latest.json`. |
| `public_content/seta_website_snippets_latest.json` | Latest public-safe website snippets consumed by the context card page. |
| `public_content/seta_website_snippets_latest.md` | Markdown companion for the latest public-safe snippets, if present. |
| `public_content/seta_website_snippets_20*.json` | Timestamped generated/archive snippets. Ignored by `.gitignore` unless explicitly included. |
| `public_content/seta_website_snippets_20*.md` | Timestamped generated/archive Markdown snippets. Ignored by `.gitignore` unless explicitly included. |

The public snippets payload must preserve:

```json
{
  "public_safe": true,
  "posting_performed": false
}
```

## Source vs generated ownership

### Source / manually edited

Usually safe to edit deliberately:

- `README.md`
- `docs/*.md`
- `dashboard_fix26_app.js`
- `dashboard_fix26_base.css`
- `dashboard_fix26_mode_manifest.json`
- `build_fix26_chart_store_payloads.py`
- `scripts/*.py`
- dashboard embed HTML files
- `seta_public_context_cards.html`

### Generated / should be regenerated

Do not hand-edit unless diagnosing:

- `fix26_chart_store_public.json`
- `fix26_chart_store_member.json`
- `fix26_screener_store.json`
- `public_content/seta_website_snippets_latest.json`
- `public_content/seta_website_snippets_latest.md`
- timestamped public snippet archives
- local Tableau AutoSync CSV/JSON outputs

### Local-only / do not commit

- API keys
- database URLs
- `.env` files
- local scratch folders
- local patch scripts
- backups
- logs
- generated draft queues
- private runtime packets

## File ownership rules for Dashboard v2 patches

Alert restoration patches should normally touch `dashboard_fix26_app.js` only. Manifest, builder, embeds, CSS, and payload schema should stay protected.

Pane redesign patches should normally touch `dashboard_fix26_app.js`, with tiny CSS changes only if necessary. Alert policy, payload builder, manifest, and generated payloads should stay protected.

Public/member mode refinement should normally use `dashboard_fix26_mode_manifest.json`, with light JS/CSS only if necessary.

Payload contract changes should involve `build_fix26_chart_store_payloads.py`, reader logic, and smoke tests only after UI needs are stable.

Public context card changes should normally touch `seta_public_context_cards.html`, the context-card smoke test, and generated public snippets only when intentionally refreshed.

## Smoke-test expectations

Before merging dashboard behavior changes:

```powershell
python scripts/smoke_fix26_dashboard.py
```

Before publishing public context card changes, verify:

- `seta_public_context_cards.html` loads locally
- `public_content/seta_website_snippets_latest.json` exists
- `public_safe` is `true`
- `posting_performed` is `false`
- no public copy makes price predictions or trade recommendations

## Local dependency notes

The refresh batch file currently assumes local Windows paths such as:

```text
C:\Users\shane\anaconda3\python.exe
C:\Users\shane\export_enriched_chart_history_v2.py
C:\Users\shane\snt_exports
G:\My Drive\Tableau_AutoSync
C:\Users\shane\sentiment-dash
```

It also expects `TWT_SNT_DB_URL` to be available in the environment.

Because those dependencies are local, repo changes should avoid making the refresh path more brittle. If the workflow is later generalized, prefer a separate setup/config phase.

## Naming cleanup candidates

Future cleanup candidates, not immediate changes:

- `interactive_dashboard_fix24_public_embed.html` could eventually be aliased or renamed for Dashboard v2.
- `interactive_dashboard_fix24_member_embed.html` could eventually be aliased or renamed for Dashboard v2.
- `fix26` filenames can remain internal build-version names while docs refer to the broader Dashboard v2 track.

Do not rename these files until there is a dedicated compatibility plan, because external embeds or links may depend on them.

## Recommended repo hygiene

- Keep `main` stable.
- Use feature branches for behavior changes.
- Keep generated payload commits separate from code refactor commits when practical.
- Bump dashboard JS cache tokens deliberately when browser cache invalidation is needed.
- Keep smoke-test expectations updated after intentional cache-token changes.
- Avoid committing local patch helpers and backups.
- Avoid committing draft content unless it is explicitly public-safe website content.
